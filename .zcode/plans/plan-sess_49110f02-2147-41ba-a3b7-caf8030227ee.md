# План: оптимизация MTProto-прокси под LTE

## Корень проблемы
LTE — lossy-сеть (потери, jitter, переменный RTT), WiFi — стабильный канал с большими буферами. Текущая конфигурация (`-a tls` CLI-режим) не оптимизирована под lossy-сети: дефолтный TCP `cubic`, маленькие буферы ядра, Docker NAT на hot path, нет MTU-probing. Это и убивает скорость на LTE, оставляя WiFi нормальным.

## Что делаем (4 направления, по ROI)

### 1. sysctl-тюнинг ядра — главный рычаг (~80% эффекта на LTE)
**Файлы:** `config/99-tg-proxy.conf` (новый), `scripts/sysctl-apply.sh` (новый)

Значения:
- `net.ipv4.tcp_congestion_control = bbr` + `net.core.default_q_disc = fq` — BBR держит скорость на lossy-линке вместо cubic'а, который «душится» на каждой потере.
- `net.core.rmem_max = 8388608`, `net.core.wmem_max = 8388608` — буферы 8 МБ (сервер 1 ГБ RAM, ≤5 пользователей, 5×3×8 = 120 МБ пиковый расход).
- `net.ipv4.tcp_rmem = 4096 131072 8388608`, `tcp_wmem = 4096 65536 8388608` — авторазмер.
- `net.ipv4.tcp_slow_start_after_left = 0` — не сбрасывать скорость после потери (важно для LTE).
- `net.ipv4.tcp_mtu_probing = 1` — автоопределение MTU, лечит «чёрные дыры» на мобильных маршрутах.
- `net.ipv4.tcp_fastopen = 3` — TFO, экономит RTT на установке соединения.
- `net.netfilter.nf_conntrack_max = 262144` (если модуль есть) — больше слотов conntrack.

`sysctl-apply.sh`: `modprobe tcp_bbr`, проверяет поддержку, копирует конфиг в `/etc/sysctl.d/`, `sysctl --system`, верифицирует что BBR активен (`sysctl net.ipv4.tcp_congestion_control`).

### 2. network_mode: host — убирает Docker NAT (~10–15%)
**Файл:** `docker-compose.yml` (правка)

- `network_mode: host` вместо `ports: ["443:443/tcp"]` (при host-network маппинг портов не нужен/игнорируется — контейнер биндит host:443 напрямую, без docker-proxy/conntrack на входе).
- Монтирование config-файла (см. п. 3).
- Пиннинг image tag к версии (ради стабильного пути монтирования config) — `latest` оставляю, путь детектит deploy.sh.

### 3. App-level tune через config-файл (полный конфиг)
**Файл:** `config/prod-sys.config` (новый, в `.gitignore`)

Генерится `deploy.sh` из шаблона — включает и tune-опции, и секцию `ports` (port/secret/tag из .env). Монтируется в контейнер, CLI-флаги из docker-compose убираются.

- `init_dc_connections = 2` (сервер 1 vCPU, 2 DC-коннекта оптимально).
- `clients_per_dc_connection = 100` (5 пользователей, с запасом).
- `upstream_socket_buffer_size = 1048576` (1 МБ).
- `downstream_socket_buffer_size = 1048576`.
- `init_timeout_sec = 60` — запас на LTE handshake с высоким RTT.
- `domain_fronting = sni` — failed fake-TLS handshake форвардится на реальный Xmax.ru (HTTP/2 200 подтверждён).
- `reset_close_socket = handshake_error` — RST на ошибочных handshake'ах, меньше TIME_WAIT.

`replay_check_*` и `allowed_protocols` — не дублируются, пусть живут на дефолтах образа (не перетрутся при обновлении).

### 4. Диагностика — найти реальный bottleneck
**Файл:** `scripts/diagnose.sh` (новый)

Показывает: активный congestion control (должен быть bbr), `ss -tin` (retransmits/rate по соединениям), счётчики conntrack, drop'ы, `docker stats`, и принятый/отправленный объём. Запуск: `bash scripts/diagnose.sh`.

### 5. deploy.sh — связать всё
**Файл:** `scripts/deploy.sh` (правка)

Дополнительно к текущему:
- Перед `docker compose up`: `ss -tlnp | grep :443` → если порт занят чем-то кроме `docker-proxy` → предупреждение (host network не взлетит).
- Шаг: применить sysctl (`bash scripts/sysctl-apply.sh`) + проверить BBR.
- Шаг: сгенерировать `config/prod-sys.config` из `.env` (полный конфиг, включая порт/секрет/tag/allowed_protocols + tune-опции).
- Убрать `command:` из docker-compose.yml — контейнер стартует с монтированным конфигом.
- После `up`: верифицировать, что опции применились (grep `allowed_protocols` в логах).
- В финальном выводе: показать активный CC, подсказку `diagnose.sh`.

### 6. Документация
- `README.md`: раздел «Оптимизация под LTE», команды применения sysctl, диагностика.
- `.env.example`: без изменений по secret (остаётся как есть); опционально комментарий про BBR.

## Порядок внедрения
1. `config/99-tg-proxy.conf` + `scripts/sysctl-apply.sh` (ядро — самый быстрый эффект).
2. `docker-compose.yml` (host network + mount).
3. `config/prod-sys.config`.
4. `scripts/diagnose.sh`.
5. `scripts/deploy.sh` (связка + верификация).
6. `README.md` / `.env.example`.

## Что НЕ трогаем
- Формат клиентской ссылки (`ee{secret}{domain_hex}`) — корректен, домен `Xmax.ru` остаётся.
- SNI-домен — это атрибут клиентской ссылки, прокси его не настраивает (клиент шлёт в ClientHello). `domain_fronting: sni` форвардит именно на этот домен, зацикливания нет (Xmax.ru ≠ 5.188.20.78).

## Ожидаемый результат
- WiFi: остаётся как есть.
- LTE: кратный прирост скорости/стабильности за счёт BBR + буферов на lossy-пути; убирается NAT-overhead; прокси устойчивее к DPI-пробам.

## Замечание по риску
Все app-level изменения аддитивны и деградируют gracefully. sysctl-изменения безопасны (стандартные сетевые тюнинги), apply-скрипт проверяет поддержку BBR ядром и откатывается на дефолт при отсутствии.
